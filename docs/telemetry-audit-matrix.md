## Initial zero-telemetry interpretation for Veil

For Veil, "zero telemetry" must eventually mean:

- no product telemetry collection enabled by default
- no telemetry upload enabled by default
- no studies/recipe enrollment active by default
- clean-profile verification proving the claim

This does not yet mean crash reporting has been removed. Crash reporting is adjacent but must be documented and handled separately because disabling it blindly can reduce diagnosability without solving the full privacy problem.

## Compile-time coverage note

`VEIL-0100` intentionally does not force `MOZ_DATA_REPORTING` fully off, because upstream derives it from `MOZ_TELEMETRY_REPORTING`, `MOZ_SERVICES_HEALTHREPORT`, `MOZ_NORMANDY`, and crashreporter. Veil disables the first three at build time and keeps crashreporter as a separate product decision.

## Current build-backed status

- `scripts/verify_telemetry_runtime.py upstream/firefox/obj-veil` passes for the current objdir and confirms:
  - `MOZ_TELEMETRY_REPORTING` is explicitly unset in Veil's mozconfig
  - `MOZ_TELEMETRY_REPORTING`, `MOZ_SERVICES_HEALTHREPORT`, and `MOZ_NORMANDY` are not defined in `config.status.json`
  - built `firefox.js` carries Veil's default-off telemetry, study, discovery, New Tab, first-run data-reporting, region, connectivity, and captive-portal prefs
  - built `greprefs.js` lacks both health-report upload prefs and `datareporting.policy.*` defaults
  - built `preferences.xhtml` confirms the removed upload/studies controls stay absent while the crash-report checkbox remains present
  - built `BrowserComponents.manifest` lacks both Normandy startup and shutdown registration entries
  - built `xpcshell` reports `AppConstants.MOZ_TELEMETRY_REPORTING === false` and `AppConstants.MOZ_NORMANDY === false`
  - `MOZ_DATA_REPORTING` remains defined and is therefore a claim boundary Veil must keep documenting
- Clean-profile startup evidence in `artifacts/network-audit/after-20260315-mozdata/summary.json` now shows that Veil no longer persists `toolkit.telemetry.cachedClientID`, `toolkit.telemetry.cachedProfileGroupID`, `datareporting/state.json`, `datareporting.dau.cachedUsageProfile*`, `toolkit.telemetry.previousBuildID`, `nimbus.profileId`, or `datareporting.policy.*` first-run state in a fresh profile.
- Veil still keeps `MOZ_DATA_REPORTING` defined at build time, so the remaining zero-telemetry caveat is now primarily compile-time/data-reporting-policy hygiene rather than a measured clean-profile identifier leak.
